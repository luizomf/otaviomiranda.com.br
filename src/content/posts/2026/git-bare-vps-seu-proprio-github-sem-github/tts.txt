Git [B EH1 R] não é "git hub".

Só pra constar, aqui vai uma comparação bem besta, mas útil.

O git é o motor.

O "git hub" e o git lab são plataformas inteiras em volta desse motor.

Quando você usa "git hub", você ganha git remoto, interface web, pull request, review, issue e Actions.

Também ganha permissões, integrações, webhooks, releases, pacotes e um monte de coisas que não fazem parte do git em si.

Quando você usa um repositório [B EH1 R] em uma VPS, você ganha basicamente git clone, git fetch, git pull, git push, S S H e hooks do git.

E é só.

A parte do S S H que confunde.

Aqui tem uma distinção importante.

Quando você faz git push do seu notebook para a VPS, quem escolhe a chave privada é o seu notebook.

A VPS não usa uma chave privada para receber esse push.

Ela recebe uma conexão S S H e verifica se a chave pública correspondente está no authorized keys do usuário git.

Então o fluxo é assim.

O notebook usa a chave privada.

A VPS confere a chave pública.

O S S H libera.

E o git recebe o push.

Isso resolve a dúvida mais comum.

Se o servidor também precisar acessar outro git remoto, por exemplo clonar um repositório privado do "git hub" dentro de um hook ou script, aí sim muda.

Nesse caso, o git rodando no servidor precisa escolher uma chave privada específica.

Vamos cobrir os dois casos.

Criando uma chave específica para a VPS.

Use um nome dedicado, como id ed dois cinco cinco dezenove vps git.

Isso cria dois arquivos.

A chave privada fica com você.

A chave pública vai para a VPS.

Agora você copia a chave pública para o servidor.

Pode ser com ssh copy id, mas no post eu mostro do jeito explícito.

Assim você entende exatamente onde ela fica.

Você lê o arquivo ponto pub, copia a linha que apareceu e cola essa linha no authorized keys do usuário git.

Depois ajusta dono e permissão para git dois pontos git, e permissão seis zero zero.

Agora teste da sua máquina local usando essa chave e Identities Only, yes.

Se entrar, beleza.

Transformando push em deploy.

Até aqui, a VPS já funciona como remoto git.

Agora vem a parte divertida.

Usar um hook para publicar os arquivos quando você der push na branch main.

No servidor, crie o diretório de deploy em barra var barra www barra meuapp barra current.

No exemplo, esse diretório pertence ao usuário git e ao grupo www data.

Se seu Nginx, Apache ou app não usa www data, ajuste para o usuário real do serviço.

Agora crie o hook post receive.

O script começa com bash em modo mais rígido, define a branch main, define o repositório [B EH1 R] e define o alvo publicado.

Quando chega um push, o hook lê três valores: revisão antiga, revisão nova e referência.

Se a referência não for a branch main, ele ignora e segue a vida.

Se for main, ele cria o diretório de destino e faz checkout forçado daquela revisão no diretório publicado.

Depois imprime uma mensagem dizendo que o deploy foi publicado, e mostra o commit.

Agora, no seu projeto local, rode git push vps main.

Você deve ver mensagens do remoto dizendo deploy publicado e commit enviado.

O post receive roda no servidor depois que o git atualiza as referências.

Por isso o script consegue saber qual branch recebeu push.

Também tem outro detalhe importante.

Hooks disparados por push rodam dentro do GIT DIR.

Em repositório [B EH1 R], isso significa que caminho relativo vira uma armadilha fácil.

Use caminho absoluto no hook.

É chato, mas é o chato que evita surpresa.

Esse hook simples tem um limite.

O hook acima é bom para aprender e para deploy simples.

Mas ele faz checkout em cima do mesmo diretório sempre.

Ou seja, em barra var barra www barra meuapp barra current.

Isso significa que arquivos não rastreados podem ficar ali entre deploys.

Às vezes isso é exatamente o que você quer, como uploads, ponto env, cache local ou arquivos gerados.

Às vezes é lixo velho.

Se você quer um deploy mais limpo, eu prefiro releases versionadas.

Cada deploy cria uma pasta nova.

Depois, o symlink current aponta para a release mais recente.

Rollback fica muito mais simples.

Se tiver um serviço rodando, você reinicia o serviço depois da troca.

Não vou colocar esse como primeiro exemplo porque ele distrai um pouco da ideia principal.

Mas, para produção, é o desenho que eu gosto mais.

O checkout forçado é simples.

Release versionada é mais disciplinada.

Troubleshooting rápido.

Se der "Permission denied publickey", verifique o S S H com modo verboso.

No servidor, confira as permissões da pasta ponto S S H e do authorized keys.

Você quer a pasta com permissão sete zero zero.

E o arquivo authorized keys com permissão seis zero zero.

Se der "Too many authentication failures", confira se o bloco do config tem Identities Only, yes.

Se o hook não rodar, confira se o arquivo post receive tem bit de execução.

Se você recebeu "refusing to update checked out branch", provavelmente empurrou para um repositório que não era [B EH1 R].

Volta duas casas e cria o repo com git init menos menos [B EH1 R].

Se o deploy publicou, mas a aplicação não lê os arquivos, olhe usuário, grupo e permissão do diretório em barra var barra www.

Metade dos problemas de "git não funciona" nessa área são S S H e permissão Unix.
