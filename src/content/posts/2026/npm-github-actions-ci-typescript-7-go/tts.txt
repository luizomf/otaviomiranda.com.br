npm e "Git Hub" Actions lembram que C I também é produção; TypeScript sete troca o motor para Go.

Quando o build recebe segredo, publica pacote e decide deploy, ele já está em produção.
Mesmo rodando num runner descartável, a consequência prática é a mesma.
Hoje isso apareceu em pacote npm recém-publicado, ação de terceiro e tag que muda de lugar.
E o pior: essa mudança pode não aparecer no diff do seu aplicativo.

A primeira história junta npm, "Git Hub" Actions e tags mutáveis como risco real de produção no C I.
Em maio, falamos de "Git Hub" Actions como superfície de produção.
Agora o delta é bem concreto.
A StepSecurity publicou, em 25 de junho, que 20 pacotes npm da Leo Platform foram comprometidos.
A leva coordenada de versões maliciosas foi publicada em 24 de junho.
Segundo a empresa, essas versões saíram dentro de uma janela de três segundos.
Elas carregavam um kit voltado a roubo de credenciais de C I e C D.
A soma dos pacotes gira em torno de treze mil e seiscentos downloads por semana.
O número não tem cara de catástrofe do ecossistema inteiro.
Mas C I com segredo por perto não precisa de milhões de instalações para virar dor de cabeça.

A StepSecurity também descreveu, no dia 24, comprometimentos em duas ações do "Git Hub".
Uma era a ação mawesome, do simonecorsi.
A outra era a semantic release action, do codfish.
Nos dois casos, ela reportou commits maliciosos empurrados para o repositório.
Também reportou tags de versão apontando para outro lugar.
Quem usava a ação por tag podia rodar código diferente sem mudar uma linha no repositório da aplicação.

Essa é a parte que costuma parecer detalhe até quebrar.
Tag é apelido mutável; hash completo de commit é endereço bem mais específico.
Em caminho crítico de build, publicação, deploy e release, isso não é preciosismo de segurança; é controle de mudança.

O contexto de Snyk sobre Phantom Gyp e Miasma lembra outra porta.
Pacote npm pode executar coisa na instalação por caminhos que não aparecem como script óbvio no package ponto JSON.
O arquivo binding ponto gyp, por exemplo, entra no fluxo de build nativo e pode virar veículo de execução.
Se isso acontece dentro de um runner com token de "Git Hub", chave de registro ou segredo de cloud, a história muda.
Com credencial de release por perto, o pacote deixa de ser só dependência.

Para reduzir o estrago, o caminho é chato e bom.
Pine ações críticas por hash completo de commit.
Reduza permissão do token do runner.
Use lockfile com disciplina e crie uma espera mínima antes de consumir versão recém-publicada em C I.
Controle scripts de instalação quando fizer sentido.
Se uma referência afetada rodou durante a janela ruim, trate segredo acessível como exposto.
Investigue antes de seguir vida.
As fontes deste bloco são StepSecurity sobre Leo Platform e sobre as ações mawesome e semantic release action.
Também entra a análise da Snyk sobre Phantom Gyp e Miasma.
